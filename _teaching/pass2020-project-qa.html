--- title: Q&A Static taint analysis for Ethereum contracts ref: pass2020-project-qa --- This document contains common questions by students and answers by the TAs.


  • The Datalog template (analyze.dl) declares the input facts "structLoad" and "structStore". Do we need to use these?
    Answer: No. You can ignore structLoad and structStore as you do not need to support structuers.

  • The project description states that we must support the message variables such as msg.sender and msg.value. What other message variables should we support?
    Answer: You need to support only msg.sender and msg.value.

  • A store input facts is given by "S10 V00 S09", where S09 is stored into V00. What is assigned to variable S10?
    Answer: Nothing is stored at S10 because it is a store statement.

  • Do we need to model the precise semantics of expressions (e.g., arithmetics)?
    Answer: No. You only need to track explicit data flow along computations. The additional examplese 8-10 in the project description illustrate this point.

  • Do we need to consider non-reachable code?
    Answer: No, we will not have test cases with non-reachable code.

  • Do we need to go back to previous statements after we identify a guard?
    Answer: Yes. After you identify a guard, you clear the taint tag of function arguments and message values, and re-analyze previous statements. We add example 11 in the project description to illustrate this point.

  • Suppose we have the following if-else branch "if (cond) {...} else {...}", where "cond" is a guard. Which branch is guarded?
    Answer: Both branches are guarded. For the true branch, the guard is "cond". For the false branch, the guard is "!cond". We add example 12 in the project description to illustrate this point
